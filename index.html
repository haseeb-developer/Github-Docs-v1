What is Git?

1. Introduction

Git is a tool that tracks changes in your files over time. It lets you save versions of your work, compare them, and go back if something breaks.

2. Why it exists

Before Git, developers often saved files as final.doc, final_final.doc, and final_final_really.doc. That created confusion and mistakes.

3. Real life analogy

Think of Git like a photo album with date-stamped snapshots. Every time you make an important change, Git saves a new snapshot for you.

4. How it works

Edit file
→
Stage changes
→
Commit
→
History saved

5. Visual learning

Version history
Save point
Rollback
Team collaboration

6. Behind the scenes

Git stores snapshots of your files in a hidden repository. It does not just save one file; it keeps a chain of commits that can be compared later.

7. Common beginner mistakes

  • Committing everything without checking changes
  • Writing bad commit messages
  • Forgetting to pull updates before pushing

8. Key takeaways

  • Git helps you track changes
  • Commits are checkpoints
  • Branches help you safely experiment
  • History makes recovery easier
Remember this:

Git is your project timeline, and every commit is a saved checkpoint you can return to anytime.

What is GitHub?

1. Introduction

GitHub is a website where people store Git repositories online. It makes sharing, collaborating, and reviewing code easier.

2. Why it exists

Git helps you save versions locally. GitHub helps you share those versions with others, keep backups, and work in teams.

3. Real life analogy

If Git is your personal notebook, GitHub is the school library where everyone can see, borrow, and improve the shared notes.

4. How it works

Push code
→
Remote repo
→
Review
→
Merge

6. Behind the scenes

GitHub stores your repository on servers and gives you features like pull requests, issues, branches, and collaboration tools.

7. Common beginner mistakes

  • Pushing directly to the main branch
  • Ignoring pull request reviews
  • Not checking whether your branch is up to date
Remember this:

Git stores your history locally. GitHub gives that history a home online and makes teamwork possible.

See what others are doing

1. Introduction

You do not have to work alone in GitHub. You can open other people's repositories, browse their files, and see what branches or pull requests are active.

2. Why it exists

Teams use GitHub to share a project, review code, and keep track of every change. Seeing what others have done helps you learn faster.

3. Real life analogy

Imagine a shared classroom project folder. You can open the folder, read the latest notes, and compare what changed from last week.

4. How it works

git clone https://github.com/username/project.git git fetch origin git branch -r git pull origin main

5. Visual learning

Open repo page
Review files
Read pull requests
Compare branches

7. Common beginner mistakes

  • Cloning a repo and editing before checking the branch
  • Ignoring upstream changes from teammates
  • Forgetting to use git pull before you start a new task
Remember this:

GitHub is where you can view the shared project. Git gives you the local tools to pull those changes onto your computer.

Install Git

1. Introduction

Before you can use Git, you need to install it on your computer and confirm it works.

2. Why it exists

Git must be available on your machine so your terminal can create repositories, track changes, and save commits.

4. How it works

git --version git config --global user.name "Your Name" git config --global user.email "you@example.com"

Configure Git

1. Introduction

Git needs your identity so it can attach your name and email to every commit you create.

2. Why it exists

This helps teams know who made each change and makes history easier to review.

4. How it works

git config --global user.name "Your Name" git config --global user.email "you@example.com"

Basic commands

1. Introduction

These are the most common commands beginners use every day: initialize, stage, commit, branch, push, pull, and clone.

2. Why it exists

Each command handles one important task in your Git workflow. Together they let you save work, share work, and review team changes smoothly.

4. How it works

git init git add . git commit -m "message" git status git branch git checkout -b feature/login git push origin feature/login git pull origin main git clone https://github.com/username/project.git

7. Common beginner mistakes

  • Using git add . without reviewing what changed
  • Trying to push before creating or switching to the right branch
  • Forgetting to run git pull before continuing work on an older branch
Example workflow:

Start with git init, write your changes, use git add ., save with git commit -m "add login feature", then share with git push origin feature/login.
